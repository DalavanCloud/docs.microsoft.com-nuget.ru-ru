# <a name="nuget-14-release-notes"></a>Заметки о выпуске 1,4 NuGet

[Заметки о выпуске NuGet 1.3](../release-notes/nuget-1.3.md) | [заметки о выпуске NuGet 1.5](../release-notes/nuget-1.5.md)

17 июня 2011 г. была выпущена NuGet 1.4.

## <a name="features"></a>Возможности

### <a name="update-package-improvements"></a>Усовершенствования пакета обновления
NuGet 1.4 предоставляет много улучшений, появившихся в команду пакет обновления, облегчая сохранять пакеты в ту же версию несколькими проектами в решении. Например при обновлении пакета до последней версии, очень часто требуется всех проектов с помощью этого пакета установки для того же verision обновления.

`Update-Package` Команда теперь становится проще:

#### <a name="update-all-packages-in-a-single-project"></a>Обновить все пакеты в одном проекте

    Update-Package -Project MvcApplication1

#### <a name="update-a-package-in-all-projects"></a>Обновить пакет во всех проектах

    Update-Package PackageId

#### <a name="update-all-packages-in-all-projects"></a>Обновить все пакеты во всех проектах

    Update-Package

#### <a name="perform-a-safe-update-on-all-packages"></a>Выполнить обновление «безопасной» во всех пакетах
`-Safe` Флаг ограничивает обновления только версии с номерами основной и дополнительной версии одного компонента. Например, если установлена версия 1.0.0 пакета и в этом канале доступны версии 1.0.1, 1.0.2 и 1.1 `-Safe` флаг обновляет пакет до версии 1.0.2. Обновления, не `-Safe` флаг и обновите его до последней версии 1.1.

    Update-Package -Safe

### <a name="managing-packages-at-the-solution-level"></a>Управление пакетами на уровне решения
До NuGet 1.4 установке пакета на несколько проектов не громоздким, с помощью диалогового окна. Он требуется запуск диалогового окна один раз в проекте.

NuGet 1.4 добавляет поддержку для установки или при удалении или обновлении пакетов в нескольких проектах, в то же время. Просто запустите, щелкнув правой кнопкой мыши решение и выбрав **управление пакетами NuGet** пункт меню.

![Диалоговое окно Управление пакетами NuGet уровня решения](./media/manage-nuget-packages-solution-dialog.png)

Обратите внимание, что в строке заголовка диалогового окна отображается имя решения, а не имя проекта.
Пакет управления теперь предоставляют список флажков со списком проектов, которые следует применить эту операцию для.

![Выбор проекта NuGet пакетов управления](./media/manage-nuget-packages-project-selection.png)

Дополнительные сведения см. в разделе [управление пакетами для решения](../tools/package-manager-ui.md#managing-packages-for-the-solution).

### <a name="constraining-upgrades-to-allowed-versions"></a>Ограничивает обновления до версии допускается
По умолчанию при выполнении `Update-Package` команды в пакете (или обновление пакета с помощью диалогового окна), обновляется до последней версии в канале. И новая поддержка обновлении всех пакетов могут быть случаи, в которых требуется заблокировать диапазон конкретной версии пакета. Например известно заранее, приложение будет работать только с 2.* версия пакета, но не 3.0 и более поздних версий. Чтобы предотвратить случайное обновление пакета 3, NuGet 1.4 добавляет поддержку ограничивают диапазон версий, которые могут быть обновлены до пакетов, вручную изменив `packages.config` текстовый файл с помощью нового `allowedVersions` атрибута.

Например, приведенный ниже показано, как заблокировать `SomePackage` пакета версии 2.0 – 3,0 (исключительно) в диапазоне.
`allowedVersions` Атрибут принимает значения с помощью [формат версии диапазон](../reference/package-versioning.md#version-ranges-and-wildcards).

```xml
<?xml version="1.0" encoding="utf-8"?>
<packages>
    <package id="SomePackage" version="2.1.0" allowedVersions="[2.0, 3.0)" />
</packages>
```

Обратите внимание, что в версии 1.4, блокировка пакета диапазону конкретной версии необходимо изменить вручную. В 1,5 NuGet планируется добавить поддержку для размещения этого диапазона через `Install-Package` команды.

### <a name="automatic-update-check-for-the-nuget-dialog"></a>Автоматической проверки наличия обновлений для диалогового окна NuGet
В некоторых версиях NuGet включать новые компоненты, выраженное через `.nuspec` файл, который не распознаются в более ранних версиях диалогового окна NuGet.
Примером является введение в версии 1.4 NuGet для [указание сборки платформы](../release-notes/nuget-1.2.md#framework-assembly-refs).
По этой причине необходимо использовать последнюю версию NuGet, чтобы убедиться, что можно использовать преимущества новейших функций пакетов.
Для выполнения обновлений в NuGet более видимым, NuGet диалоговое окно содержит логику для выделения, когда доступна более новая версия.

_**Примечание**: Проверка проводится только в том случае, если **Online** вкладку был выбран в текущем сеансе._

![Управление пакетами NuGet диалоговое окно с доступна новая версия](./media/manage-nuget-packages-update-notification.png)

Чтобы отказаться от автоматической проверки на наличие обновлений, перейдите в диалоговое окно параметров NuGet и снимите флажок **автоматически проверять наличие обновлений**.

![Параметры NuGet](./media/nuget-settings.png)

Эта функция фактически была добавлена в версии 1.3 NuGet, но не будет видима, конечно, до обновления до версии 1.3, например NuGet 1.4, стало доступно.

### <a name="package-manager-dialog-improvements"></a>Усовершенствования в диалоговом окне диспетчера пакетов
* **Имена меню улучшена**: параметры меню для запуска диалогового окна были переименованы для ясности. Пункт меню становится **управление пакетами NuGet**.
* **Области сведений отображаются последние обновления**: NuGet диалоговое окно отображает дату последнего обновления в области сведений для пакета при **Online** или **обновляет** выделенной вкладки.
* **Список тегов отображается**: Отображение диалогового окна Nuget тегов.

### <a name="powershell-improvements"></a>Усовершенствования PowerShell
* **Подпись сценариев PowerShell**: NuGet включает включения использования в средах с более строгими знаком сценариев Powershell.
* **Запрос поддержки**: консоль диспетчера пакетов теперь поддерживает запросы через `$host.ui.Prompt` и `$host.ui.PromptForChoice` команд.
* **Имена исходных пакетов**: при указании источника пакета с помощью указания имени источника пакета поддерживается `-Source` флаг.

### <a name="nugetexe-command-line-improvements"></a>Усовершенствования командной строки NuGet.exe
* **Пользовательские команды NuGet**: nuget.exe является расширяемым посредством пользовательских команд с помощью MEF.
* **Простой рабочий процесс для создания пакетов символ**: `-Symbols` флаг может применяться к обычным соглашение на основе структура папок, создание пакета символов, только включив источника и `.pdb` файлы в папке.
* **Указание нескольких источников**: `NuGet install` команда поддерживает задание нескольких источников, с помощью точки с запятой в качестве разделителя или путем указания `-Source` несколько раз.
* **Проверка подлинности прокси-сервер поддерживает**: 1,4 NuGet добавляет поддержку для запроса учетных данных пользователя при использовании NuGet через прокси, который требует проверки подлинности.
* **NuGet.exe обновить критические изменения**: `-Self` флаг теперь доступен для nuget.exe для самостоятельного обновления. `nuget.exe Update` Теперь принимает путь к `packages.config` файл и попытается обновить пакеты. Обратите внимание, что это обновление может содержать только в том, что он не будет: ** обновить, добавить, удалить содержимое из файла проекта.
** Запустите скрипты Powershell в пакете.

### <a name="nuget-server-support-for-pushing-packages-using-nugetexe"></a>Поддержка Server NuGet для принудительной отправки пакетов, с помощью nuget.exe
Простой способ размещения включает NuGet [упрощенных веб репозитория NuGet](../hosting-packages/nuget-server.md) через `NuGet.Server` пакет NuGet. С помощью NuGet 1.4 облегченная версия сервера поддерживает принудительную и удаление пакетов с помощью nuget.exe.
Последнюю версию `NuGet.Server` добавляет новый `appSetting`с именем `apiKey`. Когда ключ пропущен или оставить пустым, передача пакетов на веб-канала отключена. Установка apiKey значение (в идеале надежный пароль) включает принудительную отправку пакетов, с помощью nuget.exe.

```xml
<appSettings>
    <!-- Set the value here to allow people to push/delete packages from the server.
            NOTE: This is a shared key (password) for all users. -->
    <add key="apiKey" value="" />
</appSettings>
```

# <a name="nuget-14-release-notes"></a>Заметки о выпуске NuGet выпуск 1.4

[Заметки о выпуске NuGet 1.3](../release-notes/nuget-1.3.md) | [заметки о выпуске NuGet 1.5](../release-notes/nuget-1.5.md)

NuGet 1.4 был выпущен 17 июня 2011 г.

## <a name="features"></a>Функции

### <a name="update-package-improvements"></a>Усовершенствования пакета обновления
NuGet 1.4 вызывает массу улучшений в команду Update-Package, облегчая хранить пакеты в ту же версию для нескольких проектов в решении. Например при обновлении пакета до последней версии, это очень часто требуется все проекты, с помощью этого пакета необходимо обновить до же verision.

`Update-Package` Команды теперь позволяет:

#### <a name="update-all-packages-in-a-single-project"></a>Обновить все пакеты в одном проекте

    Update-Package -Project MvcApplication1

#### <a name="update-a-package-in-all-projects"></a>Обновить пакет во всех проектах

    Update-Package PackageId

#### <a name="update-all-packages-in-all-projects"></a>Обновить все пакеты во всех проектах

    Update-Package

#### <a name="perform-a-safe-update-on-all-packages"></a>Выполнить обновление «безопасных» во всех пакетах
`-Safe` Флаг ограничивает обновления до версии с помощью того же основного и дополнительного номера версии компонента. Например, если установлена версия пакета 1.0.0, а в канале, доступны версии 1.0.1, 1.0.2 и 1.1 `-Safe` флаг обновляет пакет до версии 1.0.2. Обновление без `-Safe` флаг бы обновите пакет до последней версии 1.1.

    Update-Package -Safe

### <a name="managing-packages-at-the-solution-level"></a>Управление пакетами на уровне решения
До NuGet 1.4 установите пакет в несколько проектов был громоздким, с помощью диалогового окна. Это обязательно, запуске диалогового окна один раз в проекте.

NuGet 1.4 добавлена поддержка установки или при удалении или обновлении пакетов в нескольких проектах, в то же время. Нужно просто запустить, щелкнув решение правой кнопкой мыши и выбрав **управление пакетами NuGet** пункт меню.

![Диалоговое окно Управление пакетами NuGet для уровня решения](./media/manage-nuget-packages-solution-dialog.png)

Обратите внимание на то, что в строке заголовка диалогового окна отображается имя решения, а не имя проекта.
Операциях пакетов теперь предоставляют список флажков со списком проектов, которым должна применяться операции.

![Выбор проекта NuGet пакетов управления](./media/manage-nuget-packages-project-selection.png)

Дополнительные сведения см. в разделе [управление пакетами для решения](../tools/package-manager-ui.md#managing-packages-for-the-solution).

### <a name="constraining-upgrades-to-allowed-versions"></a>Ограничение обновляется до версии разрешено
По умолчанию, при выполнении `Update-Package` команды в пакете (или обновление пакета с помощью диалогового окна), он будет обновлен до последней версии в веб-канала. Благодаря новой поддержке для обновления всех пакетов возможны ситуации, в которых вы хотите заблокировать пакета для указанного диапазона версий. Например может заранее известно, приложение будет работать только с версии 2.* пакета, но не 3.0 и более поздних версий. Чтобы предотвратить случайное обновление пакета до 3, NuGet 1.4 добавлена поддержка ограничить диапазон версий, которые могут быть обновлены до пакетов путем редактирования вручную `packages.config` файл с помощью нового `allowedVersions` атрибута.

Например, приведенный ниже показано, как заблокировать `SomePackage` пакета версии в диапазоне 2,0-3,0 (исключая границы).
`allowedVersions` Атрибут принимает значения с помощью [формат диапазона версии](../reference/package-versioning.md#version-ranges-and-wildcards).

```xml
<?xml version="1.0" encoding="utf-8"?>
<packages>
    <package id="SomePackage" version="2.1.0" allowedVersions="[2.0, 3.0)" />
</packages>
```

Обратите внимание на то, что в 1.4, блокировка пакету под конкретный диапазон версий должен быть вручную. В NuGet версии 1.5, мы планируем добавить поддержку для размещения этого диапазона с помощью `Install-Package` команды.

### <a name="automatic-update-check-for-the-nuget-dialog"></a>Проверка автоматического обновления для диалогового окна NuGet
Некоторые версии NuGet представлены новые функции, выражается через `.nuspec` файл, который не распознает более старых версиях NuGet диалогового окна.
Одним из примеров является введение в NuGet 1.4 для [указания сборки .NET framework](../release-notes/nuget-1.2.md#framework-assembly-refs).
По этой причине важно использовать последнюю версию NuGet, чтобы убедиться, что можно использовать преимущества новейших функций пакетов.
Чтобы обновить NuGet более видимым, NuGet диалоговое окно содержит логику для выделения, когда доступна более новая версия.

_**Примечание**: Проверка проводится только в том случае, если **Online** вкладке был выбран в текущем сеансе._

![Управление пакетами NuGet диалоговое окно с доступна новая версия](./media/manage-nuget-packages-update-notification.png)

Для отключения автоматической проверки на наличие обновлений, перейдите в диалоговое окно настройки NuGet и снимите флажок **автоматически проверять наличие обновлений**.

![Параметры NuGet](./media/nuget-settings.png)

Эта функция было фактически добавлено в версии 1.3 NuGet, но не будет видима, само собой, до обновления до версии 1.3, такие как NuGet 1.4, были сделаны доступными.

* **Подписанных скриптов PowerShell**: Включение использования в средах с более строгими подписанные сценарии Powershell включает в себя NuGet.
* **Запрос поддержки**: консоль диспетчера пакетов теперь поддерживается, запрос через `$host.ui.Prompt` и `$host.ui.PromptForChoice` команд.
* **Имена источников пакета**: указав имя источника пакета поддерживается при указании источника пакета с помощью `-Source` флаг.

### <a name="nugetexe-command-line-improvements"></a>Усовершенствования командной строки NuGet.exe
* **Пользовательские команды NuGet**: nuget.exe можно расширять посредством пользовательских команд, с помощью MEF.
* **Простой рабочий процесс создания пакетов символов**: `-Symbols` флаг может применяться к обычной соглашение на основе структуру папок, создание пакета символов, только включив источника и `.pdb` файлы в папке.
* **Указание нескольких источников**: `NuGet install` команда поддерживает задание нескольких источников с помощью точки с запятой в качестве разделителя или путем указания `-Source` несколько раз.
* **Проверка подлинности прокси-сервер поддерживает**: NuGet 1.4 добавляет поддержку для запроса учетных данных пользователя при использовании NuGet за прокси-сервером, который требует проверки подлинности.
* **NuGet.exe обновления критически важное изменение**: `-Self` теперь является флаг для nuget.exe для самостоятельного обновления. `nuget.exe Update` Теперь принимает путь к `packages.config` файл и попытается обновить пакеты. Обратите внимание, что это обновление только в том, что он не будет: ** обновление, добавление, удаление содержимого в файле проекта.
** Выполнение сценариев Powershell в пакете.

### <a name="nuget-server-support-for-pushing-packages-using-nugetexe"></a>Поддержка сервера NuGet для принудительной отправки пакетов, с помощью nuget.exe
Простой способ для узла включает в себя NuGet [упрощенных веб репозиторий NuGet](../hosting-packages/nuget-server.md) через `NuGet.Server` пакет NuGet. С помощью NuGet 1.4 облегченный сервер поддерживает принудительную и удаление пакетов с помощью nuget.exe.
Последнюю версию `NuGet.Server` добавляет новый `appSetting`с именем `apiKey`. Когда ключ опущен или оставить пустым, Отправка пакетов в веб-канал отключена. Установка apiKey значение (в идеале надежный пароль) позволяет принудительной отправки пакетов, с помощью nuget.exe.

```xml
<appSettings>
    <!-- Set the value here to allow people to push/delete packages from the server.
            NOTE: This is a shared key (password) for all users. -->
    <add key="apiKey" value="" />
</appSettings>
```

NuGet теперь поддерживается в версии-кандидата средств Windows Phone Mango.
В настоящее время средства Windows Phone не поддерживается для диспетчера таким образом установить NuGet для Windows Phone средства расширения Visual Studio, может потребоваться скачать и запустить VSIX вручную.

Чтобы удалить NuGet для Windows Phone средства, выполните следующую команду.

     vsixinstaller.exe /uninstall:NuPackToolsVsix.Microsoft.67e54e40-0ae3-42c5-a949-fddf5739e7a5
